# research-method-platform

Research method platform MVP.

The current repository already includes a runnable front end and back end, plus the first complete analysis flows for:

- `descriptive_stats`
- `regression`

Current MVP scope:

- method browsing and method detail pages
- file upload for `.csv` / `.xlsx`
- structured validation
- real descriptive statistics analysis
- real regression analysis
- unified result page rendering
- minimal DOCX report export
- `packages/ui` shared package placeholder

## Local Development

### API

```bash
cd apps/api
python -m pip install -r requirements.txt
uvicorn app.main:app --reload
```

Health check:

```text
GET http://127.0.0.1:8000/health
```

Main local API flow:

```text
GET    /api/methods
GET    /api/methods/{slug}
POST   /api/uploads
POST   /api/tasks
POST   /api/tasks/{taskId}/validate
POST   /api/tasks/{taskId}/analyze
GET    /api/tasks/{taskId}/result
POST   /api/tasks/{taskId}/export
GET    /api/exports/{exportId}/download
```

Regression input convention for the current MVP:

- dependent variable column: `outcome`
- predictor columns: start with `predictor_`
- optional control columns: start with `control_`

Example:

```csv
outcome,predictor_income,predictor_age,control_region
24,10,1,1
53,18,4,0
51,22,2,1
```

### Web

```bash
cd apps/web
npm install
npm run dev
```

If the web app needs to call the local API, create `apps/web/.env.local`:

```bash
API_BASE_URL=http://127.0.0.1:8000
```

Production build check:

```bash
cd apps/web
npm run build
```

## GitHub Pages Deployment

The `apps/web` front end can now be deployed to GitHub Pages through the workflow in `.github/workflows/deploy-pages.yml`.

Important deployment notes:

- GitHub Pages only serves the static front end. `apps/api` still needs to be deployed separately.
- The Next.js app now uses static export, and analysis results are shown on the fixed web page `/result?taskId=...` instead of a dynamic task directory route.
- The workflow will publish the `apps/web/out` folder generated by `next build`.

Before enabling the workflow:

1. In your GitHub repository settings, set GitHub Pages to use **GitHub Actions** as the publishing source.
2. Add a repository variable named `NEXT_PUBLIC_API_BASE_URL` that points to your deployed API origin, for example `https://api.example.com`.
3. If you use a custom domain or want to override the default project-site path, add an optional repository variable `NEXT_PUBLIC_BASE_PATH`.
4. Configure the API deployment with `ALLOWED_ORIGINS` so it includes your GitHub Pages origin, for example:

```bash
ALLOWED_ORIGINS=https://<your-account>.github.io
```

If the repository is a project site, the page URL can still contain `/<repo-name>`, but the browser origin used for CORS remains `https://<your-account>.github.io`.
