# 2. 架构设计原则

## 2.1 先跑通闭环，再扩能力

MVP 优先保障以下链路稳定可用：

**方法浏览 -> 模板下载 -> 数据上传 -> 数据校验 -> 自动分析 -> 结果展示 -> 报告导出**

不追求一次性覆盖全部研究方法，也不在第一版引入复杂组织协作、多租户和重型权限体系。

## 2.2 计算结果与 AI 解释分离

平台中的统计量、显著性、系数、组态路径等分析结果，必须由确定性程序计算生成。AI 只参与以下辅助环节：

- 将专业错误信息转为可理解提示
- 基于结构化结果生成解释文字
- 生成报告草稿或论文式摘要
- 生成方法说明或使用引导

AI 不直接生成虚构结果，不替代真实计算。

## 2.3 方法模块化

每一种研究方法都被视为一个独立分析模块，至少包括以下 5 类定义：

- 输入格式定义
- 数据校验规则
- 分析执行流程
- 输出结果结构
- 解释模板

这样可以在系统稳定后逐步扩展到更多方法，而无需重写主流程。

## 2.4 配置优先，硬编码其次

方法说明、模板字段、校验规则、结果展示配置优先采用配置驱动，减少页面与后端逻辑中的方法分支判断。

## 2.5 前后端职责清晰

前端负责页面、交互、上传入口、状态展示、图表展示和导出入口；后端负责文件解析、校验、任务编排、分析执行、结果持久化和导出生成。

# 3. 系统范围

## 3.1 MVP 纳入范围

MVP 阶段覆盖以下核心模块：

1. 方法库
2. 方法详情页
3. 模板下载
4. 文件上传
5. 数据校验
6. 自动分析
7. 结果展示
8. 报告导出
9. 基础任务记录

## 3.2 暂不纳入 MVP 的内容

以下内容暂不作为 MVP 架构重点：

1. 多租户架构
2. 复杂 RBAC 权限模型
3. 团队协作与共享工作区
4. 消息通知中心
5. 大规模分布式任务集群
6. 商业计费系统
7. 原生移动端
8. 多语言国际化

# 4. 总体架构概览

## 4.1 分层结构

系统采用前后端分离架构，逻辑上分为四层：

1. 表现层：Next.js Web 前端
2. 应用层：FastAPI API 服务
3. 分析层：校验器与分析引擎
4. 存储层：PostgreSQL、文件存储、日志

## 4.2 逻辑架构图

```text
用户浏览器
   |
   v
Next.js Web 前端
   |
   v
FastAPI API 服务
   |-- 方法信息服务
   |-- 上传服务
   |-- 数据校验服务
   |-- 分析任务服务
   |-- 结果服务
   |-- 报告导出服务
   |
   v
分析引擎层
   |-- 描述统计模块
   |-- 相关分析模块
   |-- 回归分析模块
   |-- t 检验 / 方差分析模块
   |-- fsQCA 模块
   |
   v
存储层
   |-- PostgreSQL
   |-- 本地文件系统 / 对象存储
   |-- 日志
```

## 4.3 MVP 部署形态

MVP 推荐采用“部署简单、内部模块化”的方式：

- `apps/web` 单独部署前端
- `apps/api` 单独部署后端
- 分析器作为后端内部模块运行
- PostgreSQL 独立部署
- 文件先落本地存储，后续切对象存储

# 5. 技术选型

## 5.1 前端

推荐技术栈：

- Next.js
- TypeScript
- Tailwind CSS
- TanStack Query
- Zustand（可选）
- ECharts 或 Recharts

选择原因：

1. Next.js 适合快速搭建内容展示与业务表单混合型应用
2. TypeScript 有利于保证接口与组件契约稳定
3. Tailwind CSS 适合在 MVP 阶段快速统一页面风格
4. TanStack Query 适合处理接口请求、缓存、轮询与失效
5. 图表库可用于展示统计结果和比较结果

## 5.2 后端

推荐技术栈：

- FastAPI
- Python 3.11+
- Pydantic
- SQLAlchemy
- Alembic
- Pandas
- NumPy
- SciPy
- Statsmodels
- Scikit-learn
- Openpyxl
- python-docx 或 reportlab

选择原因：

1. FastAPI 适合构建结构清晰的 API
2. Python 生态适合文件解析、数据清洗和统计分析
3. Pydantic 适合定义稳定的请求和响应模型
4. SQLAlchemy + Alembic 适合数据库建模和迁移
5. Pandas/Statsmodels 等库可支撑 MVP 方法分析

## 5.3 数据库与存储

- 结构化数据：PostgreSQL
- 上传文件：MVP 使用本地文件系统
- 导出文件：MVP 使用本地文件系统
- 日志：应用日志 + 错误日志

## 5.4 任务执行

MVP 阶段建议：

- 轻量分析使用同步或短耗时后台任务
- 如分析时长增长，再引入 Celery 或其他任务队列

## 5.5 AI 能力接入

AI 仅用于：

- 方法说明增强
- 错误信息润色
- 结果解释文本生成
- 报告文字草稿生成

不用于：

- 真实统计计算
- 参数结果生成
- 替代方法适用性最终判断

# 7. 核心模块设计

## 7.1 方法信息模块

职责：

- 展示方法列表
- 提供方法详情
- 提供方法适用场景、输入要求、样本要求、输出说明
- 提供模板下载与立即使用入口

核心数据来源：

- 方法元信息表
- 模板配置
- 示例文件配置

## 7.2 模板管理模块

职责：

- 按方法提供模板文件
- 提供字段级说明
- 提供示例数据

关键要求：

- 模板版本可追踪
- 模板字段与校验规则一致
- 模板下载链接稳定

## 7.3 文件上传模块

职责：

- 接收 `.xlsx`、`.csv`
- 存储原始文件
- 生成上传记录
- 与分析任务建立关联

关键要求：

- 校验文件类型和大小
- 存储原始文件名与系统文件路径
- 支持替换文件与重新上传

## 7.4 数据校验模块

职责：

- 在分析前检查文件是否满足方法要求

校验层级建议：

1. 文件级：扩展名、大小、编码、可读性
2. 结构级：列名、必填字段、样本行数
3. 类型级：数值型/分类型/布尔型/集合型是否匹配
4. 方法级：例如回归所需因变量、自变量、最小样本量等

输出要求：

- `passed`
- `errors`
- `warnings`
- `summary`
- `suggestions`

## 7.5 分析任务模块

职责：

- 创建任务
- 跟踪任务状态
- 管理校验和分析执行过程
- 记录失败原因

生命周期建议：

`created -> uploaded -> validating -> validated -> analyzing -> success|failed|canceled`

## 7.6 分析引擎模块

职责：

- 按方法分发到不同分析器
- 返回统一结构的分析结果

分析器建议：

- `descriptive_stats`
- `correlation`
- `regression`
- `t_test_or_anova`
- `fsqca`

统一接口建议：

```python
class BaseAnalyzer:
    def validate_input(self, df, config): ...
    def run(self, df, config): ...
    def normalize_result(self, raw_result): ...
```

## 7.7 结果展示模块

职责：

- 展示摘要结论
- 展示详细结果表
- 展示图表
- 展示可读解释
- 展示导出入口

统一输出建议包含：

- `summary`
- `tables`
- `charts`
- `interpretation`
- `report_payload`

## 7.8 报告导出模块

职责：

- 将结构化结果转为 PDF、Word 或 Markdown
- 记录导出文件路径和时间

MVP 建议：

- 优先支持 PDF 或 Word 二选一
- 保留 Markdown/HTML 作为内部中间格式

# 8. 前端架构设计

## 8.1 页面分层

页面建议分为：

1. 路由页面层：负责路由和布局
2. 业务容器层：负责数据请求和页面状态
3. 组件层：展示方法卡片、上传区、结果块
4. 基础 UI 层：按钮、表格、卡片、提示框

## 8.2 关键页面建议

- `/`: 首页，介绍平台定位和核心入口
- `/methods`: 方法列表页
- `/methods/[slug]`: 方法详情页
- `/analysis/[methodSlug]`: 上传与分析页
- `/tasks/[taskId]/result`: 结果页
- `/dashboard`: 历史记录页，MVP 可选

## 8.3 前端状态管理

建议：

- 服务端状态：TanStack Query
- 表单状态：React Hook Form 或原生表单方案
- 轻量全局状态：Zustand（如需要）

原则：

- 避免把所有业务状态放入全局 store
- 任务轮询、上传进度、结果获取优先通过 query 管理

# 9. 后端架构设计

## 9.1 分层建议

后端建议按以下层次实现：

1. 路由层：HTTP 接口、参数解析、状态码
2. 服务层：上传、校验、分析、导出的业务编排
3. 校验/分析层：方法专属逻辑
4. 数据访问层：数据库与文件存取
5. 基础设施层：日志、配置、异常处理

## 9.2 接口分组建议

- 方法相关
- 上传相关
- 校验相关
- 分析相关
- 结果相关
- 导出相关

## 9.3 响应结构原则

建议统一响应结构：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

失败响应建议：

```json
{
  "success": false,
  "message": "参数错误",
  "error_code": "INVALID_REQUEST",
  "details": {}
}
```

# 10. 核心数据流设计

## 10.1 浏览方法

1. 前端请求方法列表
2. 后端读取方法元信息
3. 前端展示方法卡片和筛选项

## 10.2 查看方法详情

1. 前端按 `slug` 请求方法详情
2. 后端返回方法说明、模板信息、输入要求、输出说明
3. 用户可下载模板或进入分析页

## 10.3 上传并校验数据

1. 用户上传文件
2. 后端保存文件并创建上传记录
3. 创建分析任务
4. 按方法加载字段规则并执行校验
5. 返回校验报告

## 10.4 执行分析

1. 校验通过后触发分析
2. 服务层选择对应分析器
3. 分析器读取数据并执行计算
4. 规范化输出结果并落库
5. 更新任务状态

## 10.5 查看结果

1. 前端轮询或直接请求结果
2. 后端返回统一结构结果
3. 前端渲染摘要、表格、图表和解释

## 10.6 导出报告

1. 用户选择导出格式
2. 后端使用 `report_payload` 生成文件
3. 返回下载地址并记录导出记录

# 11. 分析引擎设计

## 11.1 统一输出目标

无论底层是哪种方法，前端都应拿到统一结果结构，减少页面适配成本。

## 11.2 统一结果结构建议

```json
{
  "task_id": "task_xxx",
  "method_slug": "regression",
  "status": "success",
  "summary": {
    "title": "回归分析完成",
    "highlights": [
      "模型整体显著",
      "变量 X1 对 Y 存在显著正向影响"
    ]
  },
  "tables": [],
  "charts": [],
  "interpretation": {
    "plain_language": "模型表明 X1 增加时，Y 有上升趋势。",
    "academic_style": "回归结果显示，X1 对 Y 的影响显著且为正向。"
  }
}
```

## 11.3 MVP 方法设计建议

### 描述统计

- 输入：结构化表格数据
- 输出：均值、中位数、标准差、最值、频数等

### 相关分析

- 输入：两个及以上数值变量
- 输出：相关系数矩阵、显著性水平

### 回归分析

- 输入：一个因变量、一个或多个自变量
- 输出：模型摘要、系数表、显著性、解释度

### t 检验 / 方差分析

- 输入：分组变量 + 结果变量
- 输出：组间差异统计量、显著性、组均值

### fsQCA

- 输入：条件变量、结果变量、必要时含校准值
- 输出：必要性分析、真值表、组态结果

注意：MVP 可先提供占位能力或限制为已校准数据输入。

# 12. 数据与接口设计原则

## 12.1 数据库存储原则

数据库主要存储：

- `uploads`

- 风格：RESTful
- 编码：UTF-8

# 13. 安全、日志与测试

## 13.1 文件安全

- 限制扩展名和 MIME 类型
- 限制文件大小

# 14. 性能与演进策略

## 14.1 MVP 性能策略

- 小文件优先同步校验
- 分析过程允许短时轮询
- 图表数据尽量结构化返回，避免前端二次重算

## 14.2 后续扩展方向

1. 从本地文件切到对象存储
2. 从同步任务切到异步队列
3. 从单体模块切到独立分析服务
4. 引入用户账户与历史记录
5. 增加更多方法模块和智能推荐

# 15. 架构风险与应对

## 15.1 方法范围过大

风险：首版想覆盖过多方法，导致交付延期。  
应对：首版仅保证高频、规则清晰的方法闭环。

## 15.2 文件格式混乱

风险：用户数据质量参差不齐，校验复杂度高。  
应对：模板、字段规范、错误提示三者统一设计。

## 15.3 分析耗时增长

风险：复杂方法会拖慢请求响应。  
应对：预留任务队列与异步执行演进路径。

## 15.4 结果解释失真

风险：AI 文案可能和实际结果不一致。  
应对：解释文本必须基于结构化结果生成，不直接自由发挥。

# 16. 结论

本平台在 MVP 阶段最稳妥的技术方案是：

- 前端使用 Next.js 构建方法浏览、上传和结果展示体验
- 后端使用 FastAPI 串联上传、校验、分析和导出
- 分析器采用模块化设计
- 输出结构统一，前端按统一协议渲染
- AI 只做解释增强，不直接做统计计算

一句话总结：

**这是一个以“方法配置 + 数据校验 + 模块化分析器 + 统一结果结构”为核心的科研辅助平台架构。**
