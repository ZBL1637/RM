# 2. 规划前提

## 2.1 总体目标

在 MVP 阶段完成一个可演示、可上传、可校验、可输出结果和报告的研究方法自动分析平台。

## 2.2 MVP 闭环

必须优先跑通以下闭环：

**方法浏览 -> 模板下载 -> 数据上传 -> 数据校验 -> 自动分析 -> 结果展示 -> 报告导出**

## 2.3 首批方法范围

首批建议优先级：

1. 描述统计
2. 相关分析
3. 回归分析
4. t 检验或方差分析二选一
5. fsQCA 先做占位或受限版本

## 2.4 时间假设

以下排期以 **2026-03-16（周一）** 作为启动日期进行建议，可根据团队规模压缩或延长。

---

# 3. 里程碑原则

1. 先文档后代码，先闭环后扩展。
2. 每个里程碑都要有清晰可演示成果。
3. 每个阶段只解决一类核心问题，避免横向扩张。
4. 每个阶段结束都要保留可回滚、可验收版本。
5. 优先完成高频、规则清晰的方法，不在首版追求“大而全”。

# 5.3 M2：上传与校验主链路

时间：2026-03-26 至 2026-04-01

目标：

- 打通文件上传、任务创建、数据校验
- 建立错误提示的统一结构
- 完成至少一个方法的真实校验规则

交付物：

- `POST /api/uploads`
- `POST /api/tasks`
- `POST /api/tasks/{taskId}/validate`
- 上传页
- 校验结果展示区
- 描述统计或回归分析的模板与校验规则

验收标准：

1. 支持上传 `.csv` 与 `.xlsx`
2. 校验失败能返回结构化错误
3. 前端能展示错误字段、错误类型和修改建议
4. 校验通过后可进入“开始分析”状态

风险与依赖：

- 风险：用户文件格式差异过大，解析复杂度高
- 应对：模板先收窄，优先支持标准格式

# 5.4 M3：描述统计与相关分析

时间：2026-04-02 至 2026-04-08

目标：

- 实现首批最稳妥的分析能力

- 完成统一结果结构与图表渲染

交付物：

- 描述统计分析器

1. 描述统计可输出均值、中位数、标准差、频数等

3. 结果页包含摘要、表格、图表、解释四块内容
4. 结果结构符合 `docs/data-spec.md`

风险与依赖：

- 风险：前端结果页与后端输出结构不一致
- 应对：统一先实现一版结果协议，再做样式完善

# 7. MVP 出口标准

满足以下条件可视为 MVP 完成：

1. 至少 3 个方法可真正运行分析，建议为描述统计、相关分析、回归分析
2. 首页、方法详情、上传页、结果页完整可用
3. 数据校验能返回结构化错误并能指导用户修正
4. 结果页能展示摘要、表格、图表、解释
5. 至少支持 1 种报告导出格式
6. 核心流程经过基本测试且可稳定演示

# 9. 风险清单

## 9.1 范围膨胀

表现：过早把所有研究方法都纳入首版。  
策略：用方法优先级和里程碑冻结范围。

## 9.2 数据质量不稳定

表现：用户上传文件结构差异大。  
策略：模板先行，错误提示清晰，校验器逐层执行。

## 9.3 分析结果可信度不足

表现：解释文本与计算结果不一致。  
策略：解释文本必须从结构化结果派生。

## 9.4 任务链路复杂度过高

表现：上传、任务、校验、分析状态难以维护。  
策略：先统一状态机，再逐步增加方法和导出。

---

# 10. 结论

这份里程碑计划的核心思想是：

1. 先把文档、仓库和方法浏览做好
2. 再打通上传、校验、分析、结果的主闭环
3. 最后补回归分析、导出和演示版本
4. 将复杂方法和增强能力放到第二阶段

一句话总结：

**这不是“一次性做完所有研究方法”的计划，而是一条从可演示 MVP 稳步推进到可扩展科研平台的交付路线。**
